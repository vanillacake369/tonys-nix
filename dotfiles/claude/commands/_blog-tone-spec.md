# 핵심 원칙 (10개)

## 1. 서술체 일관

- `~다`, `~이다`, `~된다`, `~한다`, `~없다`, `~수 있다` 중심
- 종결을 *인위적으로* 변주하지 않는다. 사실의 시제와 양상에 따라 자연스럽게 결정한다
- 동일 종결 5문장 연속은 단조롭다 — 그러나 회피를 위해 의도적으로 회고형 `~었다/았다` 같은 voice 변주를 강제하지 않는다

## 2. 비유·감각 표현 전면 금지

본문에서 다음 카테고리의 표현을 사용하지 않는다.

| 카테고리 | 금지 예 | 권장 대안 |
|---|---|---|
| 시각 비유 | "흐릿하다", "선명해진다", "또렷해진다" | "정리되어 있지 않다", "명확하게 드러난다" |
| 촉각 비유 | "손에 잡힌다", "쥐어준다", "걸린다" (걸림 의미) | "이해된다", "확인된다", "차단된다" |
| 무게·탄성 비유 | "묵직하다", "단단하다", "굳건하다" | "신뢰성이 높다", "안정적이다" |
| 동작 의인화 | "못 박는다", "켜진다" (한 줄로 켠다), "내려친다" | "단정한다", "활성화된다", "적용된다" |
| 압축 격언 | "이름은 X, 알맹이는 Y", "X 가 말해주는 것은 단순하다", "이 한 줄을 잘못 외워두면…" | 평이한 사실 진술로 풀어 쓴다 |
| 마법·우주·실체 | "마법", "정수", "실체", "본질" (격언 톤), "블랙박스가 아니게 된다", "비로소", "드디어" | 삭제하고 바로 사실 |

## 3. 메타·자기지시 문장 금지

다음 패턴은 작성하지 않는다.

- "이 글에서는 ~를 다룬다"
- "지금까지 ~를 봤지만…"
- "결론적으로", "정리하자면"
- "이 사실이 의미하는 바는 크다"
- "X 가 말해주는 것은 단순하다"
- "이제 ~ 알아보자", "~ 알아볼 차례이다"
- "사족이지만", "여기서 잠깐"

예외: Why 도입부에서 *글의 약속* 한 줄 ("이 글은 ~ 를 정리한다") 은 허용. Remark 의 *요약 표 도입* 한 줄 ("본문에서 다룬 N 주제를 표로 요약한다") 도 허용.

## 4. 1인칭 경험 — Why / Remark 절에 한정

- Why 도입부 1-2 문장 — "필자는 ~ 겪었고 ~ 정리했다"
- Remark 마무리 1-2 단락 — 글을 정리한 뒤의 회고와 운영 관점 논평
- **본론 (What/How) 의 모든 H2 절에 1인칭 회고 금지** — "처음엔 ~인 줄 알았다", "한참 헤맸다" 류 X

본론에서는 *독자 시점 가정 깨기* 는 OK: "직관적으로는 X 라고 가정하기 쉽다 — 그러나 실제는 Y 다".

## 5. 데이터·코드·표 중심

본론 절의 표준 흐름:

1. 주장 한 문장
2. 코드 인용 (Stripe 식 3단: 전 1줄 / 인라인 `// ← 설명` 1-3개 / 후 1줄)
3. 표 또는 Mermaid
4. 결과 정리 한 문장

비유로 설명하지 않는다. 코드/표/사실로 설명한다.

## 6. 인과 표현 — 절제하여 사용

- 명시적 인과 ("X 때문에 Y", "X 라서 Y", "X 가 Y 를 결정한다") 는 *필요한 곳에* 사용한다
- 매 절 끝에 인과적 다리 ("이 미해결은 다음 절의 입력이 된다") 를 *강제하지 않는다*
- 가벼운 다리 한 줄 ("다음 절에서 본다") 는 글 전체에 1-2회만 사용
- 절제된 인과는 백서 voice 의 자연스러운 호흡을 만든다

## 7. 헷지 표현 금지

- "아마", "어쩌면", "혹시", "생각된다", "~인 듯", "~일 수 있다" (불확실 회피 용도)
- 모르는 것은 모른다고 사실로 쓴다: "RFC 는 이를 명시하지 않는다", "공식 문서가 없다"

## 8. 호명·aside 금지

- "여러분", "우리" (독자 호명) — 실습에서 "~가 보이는가?" 같은 자문은 허용
- "사족이지만", "여기서 잠깐", "참고로" 같은 곁가지 prefix — 제거하고 바로 본문

## 9. 한 단락 한 아이디어

- 단락 내 문장은 빈 줄 없이 이어 쓴다
- 단락 간에만 빈 줄 한 개
- 단락 길이 2-5 문장 권장, 최대 6 문장
- 새 개념은 단락당 1개만 도입하고 즉시 코드/표/예시로 구체화

## 10. 본문 이모지 — H2 끝에 한 개만

- H1 / H3 이하 헤더에 이모지 X
- 한 H2 에 이모지 1개
- 본문 (헤더 외) 에 이모지 X
- 추상 이모지 (✨, 🚀, 💡, 🤔) 보다 *내용 특화* 이모지

자세한 이모지 선택 분류는 `/blog-korean` 본문의 "H2 헤더 끝의 이모지" 절을 따른다.

# 추가 룰 (보완)

본 원칙 10개 외에 글 작성·개선 시 적용하는 보완 룰. 명세는 강제가 아닌 *권장* 이지만 위반 시 검토 대상.

## 11. 보조 설명 박스 — GitHub Alert Syntax 권장

어려운 용어·곁가지 풀이는 본문에 직접 늘어놓지 말고 GitHub Alert Syntax 로 *분리* 한다.

```
> [!NOTE]
> **AEAD 암호화란**
> Authenticated Encryption with Associated Data. 암호화와 동시에
> 무결성 검증 태그를 생성하여 변조를 탐지하는 방식.
```

지원 키워드: `NOTE`, `TIP`, `IMPORTANT`, `WARNING`, `CAUTION`. 의미별로 분리해 사용한다.

- `NOTE` — 추가 정보, 용어 풀이
- `TIP` — 권장 사항, 모범 사례
- `IMPORTANT` — 놓치면 안 되는 핵심
- `WARNING` — 흔한 함정·실수
- `CAUTION` — 운영상 위험·되돌릴 수 없는 조작

규칙:
- 한 절당 최대 2-3개 (남용 시 본문 가독성 저하)
- 첫 등장 용어를 *인라인 한 줄로 풀어주는 게 부족할 때만* NOTE 사용
- NOTE 박스 안에는 H2/H3 헤더 사용 X — 박스 첫 줄 `**용어**` 형식으로 강조

## 12. 매 H2 절 도입부 브릿지 (1-2 문장) — 권장

각 H2 절의 *첫 1-2 문장* 으로 직전 절 결과를 짧게 회수하고 이번 절 주제를 한 문장으로 선언한다.

```
✅ 도입 브릿지 (앞 절의 비대칭 키 → 이번 절의 핸드셰이크)
"앞서 비대칭 키의 동작 방향을 확인했다. TLS 는 이 방향성을
핸드셰이크 단계에서 어떻게 사용하는가."

✅ 본론으로 진입
"핸드셰이크의 목적은 두 가지다. 첫째, ..."
```

"매 절 *끝* 인과 다리 강제 금지" (원칙 6) 와 충돌하지 않는다.

- 끝 다리는 메타 문장이 누적되어 글이 무거워진다 → 강제 X, 글 전체 1-2회만
- 시작 브릿지는 절의 *호흡* 을 잡아준다 → 매 H2 권장

브릿지는 *짧게* — 한 문장이 가장 자연스럽다. 길어지면 메타 누적과 같은 문제.

**과잉 적용 방지**: 5개 H2 절에 동일 브릿지 곡선이 반복되면 정형화로 분류된다. "앞 절에서 X 했다 → 그러면 이번 절은 Y 다" 같은 동일 형식이 3회 이상 등장하면 절반은 브릿지를 *삭제하고* 본론 첫 사실로 곧장 진입한다. 글 한 편 안에서 브릿지 있는 절과 없는 절의 비율은 대략 3:2 정도가 자연스럽다.

## 13. mermaid 우선 케이스 — 표보다 시각 직관성이 큰 경우

표보다 mermaid 가 *방향성 / 계층 / 시간 흐름* 을 직관적으로 보여주는 케이스는 mermaid 우선.

| 케이스 | mermaid 타입 |
|---|---|
| 메시지 시퀀스 (프로토콜, RPC) | `sequenceDiagram` |
| 신뢰 사슬, 의존성, 객체 관계 | `classDiagram` 또는 `flowchart` |
| 상태 전이 (라이프사이클) | `stateDiagram-v2` |
| 데이터 흐름 (파이프라인) | `flowchart LR` |
| 클래스 계층 | `classDiagram` |

표가 적합한 경우는 *항목별 비교* (TLS 1.2 vs 1.3 식). 두 방식이 모두 가능할 때는 mermaid 우선.

기본 제약 (`/blog-korean` 의 mermaid 절): 글당 3-5개 최대, 노드 텍스트 3줄 이내, `<` `>` 는 `&lt;` `&gt;` escape.

## 14. 개념 설명 + 사용 시점·사례 페어 룰

절 본문이 *정의 + 메커니즘* 만 있고 "언제 / 어떻게 쓰는지" 가 빠지면 결손으로 분류한다.

각 메커니즘·옵션·기능 설명에는 다음 중 하나를 *1-2 문장* 으로 함께 둔다.

- 사용 시점: "이 메커니즘은 X 상황에 사용된다"
- 사용 환경: "Y 환경에서는 활성화하는 것이 표준이다"
- 실제 사례: "Z 프로젝트에서는 N 으로 운영한다"
- 결정 기준: "A 와 B 중에서는 ~ 일 때 A 를 선택한다"

```
❌ 개념만:
"OCSP Stapling 은 서버가 CA 의 OCSP 응답을 미리 받아 핸드셰이크에
첨부하는 방식이다."

✅ 개념 + 사용 시점/사례:
"OCSP Stapling 은 서버가 CA 의 OCSP 응답을 미리 받아 핸드셰이크에
첨부하는 방식이다. 매 연결마다 CA 까지의 추가 왕복을 절감하고
클라이언트 IP 의 CA 노출을 막는다. nginx 에서는 `ssl_stapling on`
한 줄로 활성화되며, Let's Encrypt 인증서를 사용하는 대부분의 운영
환경에서 기본값으로 켜둔다."
```

특히 *트레이드오프* 절 (성능 vs 보안, 가용성 vs 정합성) 에서는 위 룰이 강제. 단순 개념 나열은 결손.

# 글 구조 (공통)

본 명세를 따르는 글은 다음 H1 골격을 가진다.

```
# Why?

(도입 1인칭 1-2 문장 + 글의 약속 1 문장)

# What?

## [절 제목] [이모지]
...

# How?

## [실습 또는 종합 검증] [이모지]
...

# Remark

(요약 표 1개 + 핵심 사실 정리 + 1인칭 경험/운영 논평 + 인접 주제 안내)

# Reference

[^X1]: ...
```

`# 끝맺음` 대신 `# Remark` 를 사용한다. Remark 는 단순 요약이 아니라 *글을 정리한 뒤의 회고 + 운영 관점 한마디 + 인접 주제 안내* 를 모두 포함한다.

# 안티패턴 — Before → After 표

본 명세 위반 사례와 권장 수정.

| 안티패턴 | Before | After |
|---|---|---|
| 시각 비유 | "처음 정리할 때 가장 흐릿했던 X 가 비로소 손에 잡혔다" | "X 가 정리되니 명령어들이 각자의 자리에 들어왔다" |
| 격언 압축 | "이름은 SSL, 알맹이는 TLS — 이 어긋남이 입문자를 혼란스럽게 만든다" | "실제 프로토콜은 TLS 이지만 표기는 SSL 로 통용되는 상태다" |
| 동작 의인화 | "ssl_stapling on; 한 줄로 켠다" | "ssl_stapling on; 한 줄로 활성화된다" |
| 메타 요약 | "표가 말해주는 것은 단순하다 — 1.3 은 옵션을 끄집어내 버렸다" | "TLS 1.3 의 설계 방향은 안전하지 않은 선택지를 표준에서 제거하는 것이다" |
| 본론 1인칭 | "(What 절 본문) 처음엔 RSA 방향을 거꾸로 외웠다. 한참 헤맸다." | (본론에서 삭제. Remark 에서 1인칭으로 한 단락 회고) |
| 매 절 인과 다리 강제 | (모든 H2 끝에 "다음 절에서는…" 형식 다리) | 가벼운 다리는 글 전체에 1-2회만 |
| 헷지 | "아마 이 부분은 ~ 일 것이다" | "RFC 는 이를 명시하지 않는다" |
| 매 절 동형 Tension → Resolution | (모든 H2 가 "통념 → 반례 → 실제 구조" 같은 곡선) | 절마다 다른 리듬 (연표, 표 비교, 시퀀스, 라이프사이클) |

# 빠른 점검 체크리스트

작성·개선 후 다음을 본인 점검한다.

- [ ] 비유 카테고리 (시각/촉각/무게/의인화/격언) 0건
- [ ] 메타 자기지시 문장 0건 (Why 약속·Remark 요약 도입 제외)
- [ ] 본론 1인칭 0건 (Why/Remark 만 허용)
- [ ] 헷지 표현 0건
- [ ] 호명/aside 0건
- [ ] 각 H2 끝 이모지 1개, H1/H3 이모지 0개
- [ ] 한 단락 한 아이디어, 단락 길이 2-5 문장
- [ ] 매 절 끝 인과 다리 강제 X, 글 전체 1-2회 한정
- [ ] 매 H2 절 *도입부* 브릿지 1-2 문장 (권장)
- [ ] 어려운 용어는 GitHub Alert 박스 (`[!NOTE]`) 로 분리, 절당 2-3개 이내
- [ ] 메시지 시퀀스·신뢰 사슬·상태 전이는 mermaid 우선
- [ ] 트레이드오프·옵션 설명에 사용 시점·사례 페어 (강제)
- [ ] 코드 블록 Stripe 3단 맥락 (전/내/후)
- [ ] 표 우선, ASCII 박스 X

# 갱신 절차

본 명세를 갱신할 때 다음을 따른다.
