# /tech-blog-korean — 한국어 기술 블로그 작성

사용자가 요청한 주제: **$ARGUMENTS**

위 주제에 대해 한국어 기술 블로그 글을 다음 워크플로우로 작성한다.

만약 `$ARGUMENTS` 가 비어 있다면, 먼저 _주제와 출처 자료_ 를 사용자에게 물어본 뒤 진행한다.

---

# 작성 워크플로우 (8 단계)

## 1단계 — 주제와 출처 확인

다음을 사용자에게 짧게 묻는다.

- 참고할 출처 자료 (강의 노트, 공식 docs, 소스 코드 링크)
- 이미 작성된 Why? 섹션이 있는지

본문 작성을 _시작하지 않는다_.

## 2단계 — 내러티브 스켈레톤 + 상세 TOC 제안

다음을 포함한 **내러티브 스켈레톤** 과 TOC 를 만든다.

### 핵심 질문

이 글 전체를 관통하는 _단 하나의 질문_ 을 정한다.

### 절 인과 사슬 (Tension → Resolution)

각 절이 다음 절로 넘어가는 인과관계를 설계한다.
절 순서는 **Outside-In** 방향을 기본으로 한다: 독자가 관찰 가능한 현상(외부)에서 시작하여 원인(내부)으로 파고든다.

```
절 A: [제목]
  → 독자의 기존 가정: "당연히 ~일 것이다" (뒤집힐 통념)
  → 가정이 합리적인 이유: 왜 독자가 그렇게 생각하는지 (1-2문장)
  → Tension: 이 가정이 깨지는 반례/현상/데이터
  → Resolution: 실제 구조 — 기존 가정 대신 무엇이 참인지
  → 새로운 미해결: resolution 이 열어주는 다음 질문
  → 다음 절과의 인과: 이 미해결이 왜 다음 절의 입력이 되는지

절 B: [제목]
  → ...
```

**Outside-In 순서 원칙:**
- ❌ 저자/구현 순서: 커널 → 시스템콜 → 라이브러리 → 앱
- ✅ 독자/관찰 순서: 앱 동작 → "왜?" → 라이브러리 → 시스템콜 → 커널

기존 자료의 절 순서 변경, 합치기, 분리가 필요하면 제안한다.

### Why 섹션 구조 — 문맥 있는 내러티브 도입

뜬금없이 상황을 던지지 않는다. 논문 서론이나 기술 블로그 intro처럼 문맥을 쌓아 문제를 드러낸다.

다음 흐름을 따른다.
1. **가정 설정**: "~라는 시스템을 설계한다고 가정해보자" — 독자가 겪는 구체적 상황
2. **자연스러운 설계**: "이때 다음과 같이 구성할 수 있을 것이다" — 코드 또는 다이어그램
3. **문제 발견**: "이 구조에는 큰 문제가 숨어 있다" — 설계의 결함을 드러냄
4. **문제 명명 + 배경**: 공식 명칭과 왜 근본적으로 피할 수 없는지 (footnote로 출처 연결)
5. **핵심 개념 표** (주제 — 해결하는 문제 매핑)
6. **글의 약속**: "이 글은 이 문제를 해결하는 N가지 방법을 비교한다"

```
❌ 뜬금없는 시작:
"주문을 DB에 저장하고 이벤트를 Kafka로 발행하는 코드를 작성한다."

✅ 문맥 있는 시작:
"전자상거래 시스템을 설계한다고 가정해보자.
고객이 주문을 넣으면 주문 서비스는 DB에 저장하고 결제 서비스에 이벤트를 보내야 한다.
이때 다음과 같이 코드를 작성할 수 있을 것이다. (코드)
이 구조에는 큰 문제가 숨어 있다. ..."
```

### 실습 매핑

각 실습이 What 의 어떤 절의 어떤 주장을 검증하는지 명시한다.

```
인라인 실습 ①: [제목]
  → 검증 대상: What 절 X 의 "주장 A"
  → 환경: Multipass / Vagrant / 로컬
  → 핵심 명령어 2-3개

종합 실습 (How): [제목]
  → 검증 대상: 여러 절의 개념을 조합한 종합 시나리오
```

### TOC 형식

- 모든 절 제목을 _표준 패턴_ 으로 작성 (아래 "절 제목 패턴" 참고)
- 각 H2 헤더에 _내용 특화 이모지_ 후보 (아래 "H2 헤더 끝의 이모지" 참고)
- 각 절의 한 줄 요약 + 인과 전환
- mermaid 구조도 위치 (전체 구조를 한 눈에 보여줄 때)
- 인라인 실습 위치 + How 종합 실습 미리보기

## 3단계 — 3 가지 스코프 질문

본문 작성 전 반드시 다음 3 가지를 확정한다.

| 질문             | 옵션                                                           |
| ---------------- | -------------------------------------------------------------- |
| 1. 커버리지      | (A) 전체 주제 / (B) 핵심만 / (C) 일부 + 후속글                 |
| 2. 코드 깊이     | (A) 함수 레벨까지 / (B) 구조체 정의만 / (C) 핵심 함수 1-2 개씩 |
| 3. Why 보강 정도 | (A) 주신 텍스트만 / (B) + 배경 한 단락 / (C) + 일반론까지      |

사용자 응답을 기다린다.

## 4단계 — 분량 평가와 청크 분할 결정

스코프가 확정되면 _예상 총 분량_ 을 추정한다.

### 분량 추정 규칙

- 주요 절 1 개 ≈ 100-150 줄 (코드 블록, 표, mermaid 포함)
- How? 절 ≈ 200-400 줄 (실습 5 개)
- Why + Reference ≈ 50-100 줄

```
예상 총 줄 수 = (주요 절 수 × 125) + 300 ± 100
```

### 청크 분할 트리거

다음 중 하나라도 해당하면 _분할을 제안_ 한다.

- 예상 총 분량 > 1500 줄
- 주요 도메인이 5 개 이상이고 각각이 _독립적_
- 사용자가 "시리즈", "여러 편", "나눠서" 라고 명시
- TOC 가 자연스럽게 _추상 계층_ 으로 나뉨 (예: 커널 → 노드 → 클러스터 → 네트워크)

### 분할 제안 형식

분할이 권장될 경우 다음 형식으로 제안한다.

```
이 글은 약 N 줄 예상이다.

다음 K 개 파일로 분할하는 것을 제안한다.

| 편 | 제목 | 예상 분량 | 다루는 범위 |
|---|---|---|---|
| 1 | ... | ~XXX 줄 | ... |
| 2 | ... | ~XXX 줄 | ... |
| ... | | | |

분할 시 각 편은 독립적으로 작성하되, 인접 편으로의 다리 문장으로
연결한다.

진행 방식을 선택해주세요:
(A) 단일 파일로 작성
(B) K 개 파일로 분할 + 병렬 작성 (권장)
(C) 다른 분할안 제시
```

사용자 선택을 기다린다.

분할이 결정되면 _각 편의 개별 TOC_ 를 추가로 만들어 사용자에게 확인받는다.

## 5단계 — 샘플 절 작성

대표 절 한 개를 _완전한 형태로_ 작성한다 (모든 포맷, footnote, 코드, 표, H2 이모지 포함).

분할이 결정된 경우, _첫 번째 편의 가장 핵심적인 절_ 을 샘플로 작성한다.

샘플 작성 시 명시한다.

- 어느 절인지
- 작성에 포함된 요소 (footnote 수, mermaid 유무, 표 개수, 코드 블록 수, H2 이모지)

_전체 본문을 첫 턴에 절대 작성하지 않는다_.

## 6단계 — 명시적 승인 게이트

샘플을 보여준 후 다음 형식으로 사용자 승인을 요청한다.

```
위 샘플의 톤과 포맷을 확인해주세요.

다음 항목 중 조정이 필요한 부분이 있는지 검토해주세요.

- [ ] 톤 (다/이다 체, 비유 표현 없음)
- [ ] 절 제목 패턴
- [ ] H2 헤더 이모지 (내용 특화, 추상 X)
- [ ] 코드 인용 분량
- [ ] 표 사용
- [ ] mermaid 위치와 내용
- [ ] footnote 밀도와 출처 분배
- [ ] 어려운 용어의 인라인 풀이 정도
- [ ] 다리 문장의 흐름

이대로 진행해도 될지 또는 조정할 부분이 있는지 알려주세요.
```

사용자가 _명시적으로 승인하기 전까지_ 7단계로 진행하지 않는다.

## 7단계 — 저장 위치 확인

승인을 받으면 본문 작성 전에 _저장 위치와 파일명_ 을 묻는다.

```
전체 본문을 작성하기 전, 저장 위치를 확인하고 싶습니다.

기본 옵션:
- 디렉토리: /mnt/user-data/outputs/
- 파일명 (단일): <topic>-<descriptor>.md
- 파일명 (분할 시): <series>-<NN>-<topic>.md

다음 중 하나를 선택해주세요.
(A) 기본 옵션 그대로 사용
(B) 다른 디렉토리/파일명 지정 (구체적으로 알려주세요)

분할 작성인 경우 시리즈 prefix 도 함께 알려주세요 (예: "valkey-internals", "k8s-series").
```

응답을 받아 정확한 경로를 확정한 후 8단계로 진행한다.

## 8단계 — 본문 작성과 파일 생성

저장 위치가 확정되면 본문을 작성한다.

### 단일 파일 (분량 < 1500 줄)

순차적으로 전체 본문을 작성하고 한 파일로 저장한다.

- 파일 경로: 7단계에서 확정된 경로
- 작성 후 사용자에게 파일 제시
- 분량/구조/footnote 수 요약 제공

### 분할 작성 (병렬 처리)

청크가 결정된 경우 _Task 서브에이전트_ 를 활용해 _각 편을 병렬로 작성_ 한다.

병렬 처리 절차는 아래 "병렬 작성 가이드" 절을 따른다.

# 병렬 작성 가이드 (분할 시)

긴 본문을 효율적으로 작성하기 위해 _Task 도구의 general-purpose 서브에이전트_ 를 활용한다.

## 병렬 작성 원칙

- 각 편을 _독립된 서브에이전트_ 가 작성
- 모든 서브에이전트는 _동일한 메서드_ 와 _동일한 톤_ 을 따름
- 한 메시지에서 _여러 Task 호출을 동시에 발행_ 하여 병렬 실행
- 메인 에이전트는 결과를 수집하고 _편 간 다리 문장 정합성_ 을 검증

## 서브에이전트 프롬프트 템플릿

각 서브에이전트에 다음 정보를 모두 전달한다.

```
[메서드 컨텍스트]
- 한국어 기술 블로그 작성 메서드 (이 command 파일 전체 내용)

[글 전체 정보]
- 시리즈 제목: ...
- 전체 편 수: K
- 전체 시리즈 TOC: ...
- 사용자가 확정한 스코프 (커버리지 A/B/C, 코드 깊이 A/B/C, Why 보강 A/B/C)

[이 편의 정보]
- 편 번호: N
- 편 제목: ...
- 이 편의 상세 TOC (H2 이모지 포함): ...
- 이 편의 예상 분량: ~XXX 줄

[인접 편과의 다리]
- 직전 편 (N-1) 의 마지막 결론: ... → 이 편의 첫 문장이 받아야 함
- 다음 편 (N+1) 의 첫 주제: ... → 이 편의 마지막 문장이 예고해야 함

[톤 참고]
- 사용자가 승인한 샘플 절 (5단계의 샘플 그대로): ...

[출력]
- 파일 경로: <path>/<series>-<NN>-<topic>.md
- 작성 완료 후 파일 생성하고 경로 반환
```

## 병렬 호출 예시

```
한 메시지 안에서 여러 Task 호출을 동시 발행:

Task(subagent_type="general-purpose", prompt="[편 1 작성 지시]")
Task(subagent_type="general-purpose", prompt="[편 2 작성 지시]")
Task(subagent_type="general-purpose", prompt="[편 3 작성 지시]")
Task(subagent_type="general-purpose", prompt="[편 4 작성 지시]")

동시에 4 개 서브에이전트가 작업하여 시간을 1/K 로 단축.
```

## 병렬 작업 후 검증

서브에이전트들의 결과가 모두 돌아오면 메인 에이전트가 다음을 점검한다.

- 각 편의 첫 문장이 _직전 편의 결론_ 을 받는지
- 각 편의 마지막 문장이 _다음 편_ 을 예고하는지
- 절 제목 패턴 일관성
- H2 이모지 일관성 (모든 H2 에 있는지, 내용 특화인지)
- footnote prefix 통일 (예: K1-K20 → 편 별로 K1-K15, K16-K30 등 분배 또는 편 별 prefix 따로)
- 본문 톤 일관성

문제가 있으면 _해당 편을 다시 서브에이전트에 의뢰_ 하여 수정한다.

## Mandatory: Reference Integrity

- 모든 footnote URL은 실제 접근 가능한 출처여야 함 (추론으로 URL 생성 금지)
- 출처를 찾을 수 없는 주장은 "출처 미확인 — 검증 필요"로 표기
- hedging 표현 금지 (위 톤 규칙에 따름)
- 기술적 주장에는 반드시 출처 footnote 연결

## 최종 파일 제시

모든 편이 검증 완료되면 _모든 파일을 한 번에 제시_ 한다.

```
present_files([
    "<path>/<series>-01-...md",
    "<path>/<series>-02-...md",
    "<path>/<series>-03-...md",
    "<path>/<series>-04-...md",
])
```

각 편의 분량/구조/footnote 수 요약 표를 함께 제공한다.

```
| 편 | 제목 | 줄 수 | mermaid | footnote |
|---|---|---|---|---|
| 1 | ... | XXXX | X 개 | XX 개 |
| 2 | ... | XXXX | X 개 | XX 개 |
| ... | | | | |
| 합계 | | XXXX | XX 개 | XX 개 |
```

# 절 제목 패턴 (필수)

모든 What? 절 제목은 다음 패턴을 따른다.

```
[주제] — [핵심 내부 구조] (을/를/와/과) (활용한/결합한/구현한/사용한) [역할 또는 정의]
```

## 좋은 예 (이모지 포함)

- `String — SDS 자료구조와 세 가지 인코딩을 활용한 key-value 저장소 💾`
- `List — quicklist 와 listpack 을 결합한 양방향 시퀀스 📋`
- `Sorted Set — skiplist 와 hashtable 을 결합하여 구현한 점수 정렬 집합 🏆`
- `Namespace — task_struct 의 nsproxy 를 통해 분리한 프로세스 시야 👁️`
- `etcd — Raft 합의 위에 구현한 strong-consistency KV 스토어 🗳️`
- `kube-proxy — netfilter NAT 규칙으로 구현한 Service 가상 IP 라우팅 🔀`

## 금지 예

- `String — SDS 라는 작은 우주` (비유)
- `etcd — Valkey 의 심장` (비유)
- `1.1 String 데이터 타입` (번호 매김)
- `Background and Overview` (모호, 영어)
- `String 살펴보기` (내부 구조 + 역할 부재)

# H2 헤더 끝의 이모지

모든 `##` (H2) 헤더의 끝에 _내용을 상징하는 이모지 한 개_ 를 추가한다.

H1 (`#`) 헤더와 H3 (`###`) 이하 헤더에는 이모지를 사용하지 않는다.

## 이모지 선택 원칙

- 절의 _핵심 개념_ 을 시각적으로 표현
- 한 헤더에 _이모지 하나만_ (중복 금지)
- _추상적/일반적 이모지_ (✨, 🚀, 💡, 🤔) 보다 _내용 특화 이모지_ 우선
- 헤더 텍스트 마지막 단어 뒤에 _공백 한 칸_ 후 배치
- 한 글 안에서 _같은 이모지 중복 사용 최소화_ (절마다 다른 이모지)

# 톤 규칙 (엄격)

## 따른다

- `다 / 이다` 체 (서술체)
- 직접적, 서술적 산문
- _Italic_ — 개념 강조: `*pure function*`, `*amortized O(1)*`
- **Bold** — 정의되는 용어: `**derivation**`, `**namespace**`
- > Blockquote — 핵심 takeaway 만 (절당 1-2 개 최대)
- H2 헤더 끝의 이모지 (위 "H2 헤더 끝의 이모지" 참고)

## 금지

- `네요 / 예요 / 죠` 어미
- 독자 직접 호명 (`여러분`, `우리`) — 실습에서 "~가 보이는가?", "~를 확인해보자" 는 허용
- 비유적/과장 본문 표현
    - `작은 우주`, `정수`, `심장`, `마법`, `우아한 트릭`, `진짜 흥미로운`
    - `여기서 한 단계 더 들어가면`, `놀랍게도`, `사실은`
    - `블랙박스가 아니게 된다`, `비로소 ~가 된다`, `드디어`
    - `이 사실이 의미하는 바는 크다` — 삭제하고 바로 내용으로
    - `~의 정체다`, `~의 실체다` — "~에 해당한다", "~이 확인된다" 로 순화
    - `결국 마주하게 되는 질문은 하나다` — 삭제하고 바로 질문으로
- 위트가 전체의 0.5% 초과
- 헷지 표현: `아마`, `어쩌면`, `혹시`, `생각된다`
- 과한 강조 (한 단락에 italics 1-2 개 초과)
- 본문 안의 이모지 (H2 헤더 끝 외에는 사용 금지)

# 포맷 규칙

## 문단 구성

### 원칙
- **한 문단 = 한 주제** (single idea per paragraph)
- 문단 내 문장은 빈 줄 없이 이어 쓴다
- 문단 간에만 빈 줄 한 개로 분리
- 문단 첫 문장은 topic sentence (이 문단이 말하려는 것)
- 문단 마지막 문장은 다음 문단으로의 인과 전환 또는 요약

### 문단 길이
- 권장: 2-5 문장 (디지털 읽기 최적)
- 최대: 6 문장 (초과 시 주제가 두 개일 가능성 — 분리 검토)
- 최소: 1 문장 문단 허용 (강조/전환 목적에 한정, 연속 2개 금지)

### 문단 내 응집 (cohesion)
- Old-to-New: 각 문장은 직전 문장의 정보를 받아 새 정보를 추가
- 접속 표현으로 논리 연결: "~이므로", "왜냐하면 ~", "따라서", "그 결과"
- 같은 주제의 문장을 빈 줄로 분리하지 않는다

### 문단 간 전환 (coherence)
- 새 주제/새 관점이 시작될 때만 빈 줄로 분리
- 절 전환과 동일한 인과적 다리 원칙 적용

### 문장 길이 변주
- 짧은 문장(15단어 이하)과 긴 문장(25단어 이하)을 섞어 리듬 형성
- 같은 길이의 문장이 3개 이상 연속되면 단조로움 — 변주 필요

### 정보 밀도 제어
- 새 개념은 **문단당 1개만** 도입
- 도입 직후 반드시 예시/코드/표 중 하나로 구체화
- 2개 이상 새 개념이 한 문단에 등장하면 분리 대상
- 추상 → 구체 교대 리듬: 개념 문단 → 코드/표 → 개념 문단 → 코드/표

## 표 — ASCII 보다 우선

다음의 경우 _반드시_ 마크다운 표.

- 메모리 레이아웃, 임계 설정, 인코딩 비교, 복잡도 비교, 필드 정의

ASCII 박스 그림 _금지_.

## 코드 블록 (Stripe식 3단 맥락)

모든 코드 블록은 **전(맥락) → 내(하이라이트) → 후(takeaway)** 구조를 따른다.

### 코드 전 — 맥락 한 줄
코드 블록 직전에 "이 코드가 보여주는 것"을 한 문장으로 명시한다.

### 코드 내 — 인라인 하이라이트
핵심 라인에 `// ← 설명` 주석으로 독자의 시선을 유도한다.

### 코드 후 — takeaway 한 줄
코드 블록 직후에 "핵심 포인트"를 한 문장으로 정리한다.

### 예시
```
다음 코드는 ziplist에서 listpack으로 전환되는 임계점을 보여준다.

\`\`\`c
// src/t_list.c
void listTypeTryConvertListpack(robj *o) {
    if (o->encoding == OBJ_ENCODING_LISTPACK) {
        if (lpLength(o->ptr) > server.list_max_listpack_size)  // ← 이 조건이 전환을 결정
            listTypeConvert(o, OBJ_ENCODING_QUICKLIST);
    }
}
\`\`\`

`list_max_listpack_size` 를 초과하는 순간 인코딩이 quicklist 로 전환된다.
```

### 기본 규칙
- Triple-backtick + 언어 hint
- 단순화한 의사코드는 `// 단순화` 또는 `// (핵심부)` 명시
- 실제 소스 경로 주석: `// src/sds.c`
- 한 코드 블록 약 25 줄 이내, 길면 분할
- 인라인 하이라이트(`// ←`)는 블록당 1-3개 (과하면 노이즈)

## Mermaid

- 글당 3-5 개 최대
- _개념 전환점_ 에만
- `<` `>` 는 `&lt;` `&gt;` 로 escape
- 노드 텍스트 최대 3 줄

# 초보자용 용어 풀이

처음 접하는 독자에게 어려울 수 있는 용어는 _인라인으로 풀어준다_.

## 패턴

1. 문맥 안에서 용어 도입
2. 한국어 번역 제공 (있다면)
3. 짧은 시나리오나 예시
4. 작은 표나 리스트로 시각화
5. 이후로는 자신 있게 사용

## 풀어줄 대상

- 흔하지 않은 CS 용어: `amortized`, `lock-free`, `copy-on-write`, `idempotent`, `radix tree`
- 도메인 특화 jargon (첫 등장 시)
- 흔하지 않은 약어 (CPU/GPU/OS/IP 는 풀이 불필요)
- 내부 개념을 참조하는 설정명

## 풀지 않을 대상

- 이전 절에서 이미 풀어준 용어
- 흔한 용어 (loop, function, array)
- 기술 독자에게 기본인 프로그래밍 개념

`[^X1]: 설명. <https://full-url>`

`X` 는 글 내 일관된 한 글자 prefix (Valkey = V, K8s = K, NixOS = N).

분할 작성의 경우 _각 편마다 다른 prefix_ 를 사용해 충돌 방지 (편 1 = K1, 편 2 = N1 등 또는 K1-K20, K21-K40 등 범위 분할).

## 소싱

WebSearch 등을 활용하여 적합한 참고자료를 적극적으로 찾는다.
Why 섹션의 문제 정의, 각 패턴/개념의 원전, 구현체의 공식 문서는 반드시 footnote로 연결한다.

## 사용 대상

- 패턴/개념의 공식 명칭 첫 등장 → 원전 또는 공식 문서
- 핵심 개념 (CAP, ACID 등) → 논문 또는 위키피디아
- 구현체 (Debezium, Dapr 등) → 공식 documentation
- 소스 코드 참조 (파일 경로 + 라인 범위 필수, 예: `#L42-L58`)
- 외부 권위 출처 (논문, KEP/RFC, 디자인 문서)
- 한국어 기술 참고 사이트 (redisgate.kr, kubernetes.kr, devocean.sk.com 등)

## 절대 사용 금지

- 일반 상식
- 글 내에서 자체 정의한 용어
- 예시나 가상 시나리오

## 분배

- 주요 절당 6-9 개 footnote
- _최소 2 종류_ 혼합: 공식 소스 / 공식 docs / 한국어 참고

## URL 컨벤션

- `<angle brackets>` 로 감싸기
- GitHub 소스: `https://github.com/<org>/<repo>/blob/<branch>/<path>#L<start>-L<end>`
- 소스 코드 참조 시 **반드시 라인 범위를 포함** (예: `#L42-L58`)
- 라인 범위가 1줄이면 `#L42`, 여러 줄이면 `#L42-L58`
- 가능한 경우 URL 존재 검증 후 사용

# 절 간 전환과 개연성 (필수)

## 절 내부 리듬 — Tension → Resolution

각 절 내부도 미니 내러티브 구조를 따른다. 독자의 기존 이해를 존중하면서 자연스럽게 뒤집는다.

```
[① 기존 가정 소개]
독자가 당연하게 여기는 통념을 먼저 제시한다.
이 가정이 합리적인 이유를 1-2문장으로 설명한다 — 독자의 사고를 대변하여
"이렇게 생각하는 것이 자연스럽다"는 공감의 문맥을 깔아준다.

[② Tension — 가정이 깨지는 순간]
그런데 실제로 동작을 관찰하면, 혹은 코드를 추적하면,
이 가정과 다른 결과가 나타난다.
구체적인 반례/데이터/코드로 가정의 균열을 보여준다.

[③ Resolution — 실제 구조]
가정이 왜 틀렸는지, 실제로는 어떤 구조인지를 설명한다.
이 과정에서 독자의 mental model이 업데이트된다.

[④ 새로운 미해결 → 다음 절로]
resolution이 열어주는 새 질문을 제시하여 다음 절의 입력으로 넘긴다.
```

**핵심: 물 흐르듯 바느질한다.**

각 단계 사이에 "왜 이 생각이 자연스러운지", "왜 이것이 문제인지", "이것을 어떻게 해결할지"를 명시적으로 연결한다. 독자가 문맥을 놓치지 않도록 이전 맥락을 한 문장으로 회수한 뒤 새 정보를 추가한다(Old-to-New 원칙).

❌ 끊어지는 가정 뒤집기:
```
Redis 는 단일 스레드이다. 그런데 실제로는 I/O 스레드가 있다.
```

✅ 문맥을 바느질한 가정 뒤집기:
```
Redis 가 단일 스레드로 동작한다는 것은 널리 알려진 사실이며, 실제로 명령어 실행은 하나의 메인 루프에서 직렬 처리된다. 이 설계 덕분에 lock 없이도 데이터 정합성을 보장할 수 있다. 그렇다면 네트워크 I/O 도 메인 스레드가 처리하는가? Redis 6.0 이전에는 그랬다. 그러나 클라이언트 수가 증가하면 read/write syscall 이 병목이 되며, 이를 해결하기 위해 6.0 에서 I/O 스레드가 도입되었다. 핵심은 명령어 실행은 여전히 단일 스레드이되, I/O 만 병렬화한다는 점이다.
```

## 절 전환 — 인과적 다리 (기계적 연결 금지)

모든 절의 끝에는 _아직 모르는 것_ 과 _그것을 알아야 하는 이유_ 를 명시하여 다음 절로 넘긴다.

❌ 기계적 전환:
```
다음 절에서는 X 를 본다.
```

✅ 인과적 전환:
```
여기까지 X 의 구조를 확인했다. X 가 이렇게 동작한다면, 자연스럽게 "Y 는 어떻게 처리되는가?"라는 질문이 생긴다. 그런데 X 만으로는 Y 가 해결되지 않는다 — X 의 scope 가 단일 노드에 한정되기 때문이다. Y 를 해결하려면 노드 간 통신 계층인 Z 를 이해해야 하며, 다음 절에서 이를 다룬다.
```

## 문장 간 개연성 — 뚝뚝 끊기지 않게

인접한 문장 사이에 _왜 그런지, 그래서 어떻게 되는지_ 를 연결한다.

❌ 끊어지는 서술:
```
NixOS 는 모든 바이너리를 /nix/store 에 저장한다.
/usr/bin/socat 같은 경로에는 아무것도 없다.
kubeadm 은 이 경로를 하드코딩으로 기대한다.
```

✅ 연결된 서술:
```
NixOS 는 모든 바이너리를 /nix/store 에 저장하며, /usr/bin/socat 같은 FHS 표준 경로에는 아무것도 두지 않는다. 이것이 NixOS 의 순수함을 보장하는 원리이지만, K8s 에서는 문제가 된다. 왜냐하면 kubeadm 의 preflight check 가 이 FHS 경로를 하드코딩으로 기대하기 때문이다.
```

연결에 사용하는 표현:
- "~이므로", "~하기 때문이다", "왜냐하면 ~"
- "왜일까? ~해야 하는 이유는 ~때문이다"
- "~하지 않은 채 ~할 수도 있다. 그러나 ~한다면"
- "따라서", "그 결과"

## 실습 브릿지 — 왜 이 시점에 이걸 해야 하는지

❌ 기계적:
```
직전 절에서 X 라고 했다. 확인한다.
```

✅ 개연성 있는 도입:
```
지금까지 X 라는 개념을 다뤘지만, 실제로 어떻게 생겼는지는 직접 확인하지 않으면 추상으로 남는다. 이 실습에서는 ~를 통해 ~를 직접 관찰하고, ~가 실제로 ~하는지 검증한다.
```

## 실습 안 개념 회수

명령어 결과를 보여준 뒤, 해당 절의 개념과 연결한다.

```
inode 번호가 동일한 것이 보이는가?
namespace 절에서 다룬 "같은 namespace 에 속하면 같은 inode 를 공유한다"는 원리가 확인된다.
```

회수는 절당 1-2 회로 제한한다. 과하면 오글거린다.

## 분할 작성 시의 편 간 다리

분할 작성에서는 각 편의 _시작과 끝_ 도 다리 문장이 필요하다.

- 편 N 의 Why? 첫 문장 → 편 N-1 결론 받기
- 편 N 의 끝맺음 마지막 문장 → 편 N+1 예고

이 정합성은 8단계 검증 시 메인 에이전트가 점검한다.

# 빠른 점검 체크리스트

## 워크플로우 진행 점검

- [ ] 1단계: 출처 확인 마침
- [ ] 2단계: TOC 사용자 확인
- [ ] 3단계: 3 가지 스코프 확정
- [ ] 4단계: 분량 평가 + (필요 시) 분할 결정
- [ ] 5단계: 샘플 절 작성 완료
- [ ] 6단계: _명시적 승인_ 받음
- [ ] 7단계: 저장 위치 확인됨
- [ ] 8단계: 본문 작성 (단일 또는 병렬)

## 본문 품질 점검

### 구조
- [ ] 제목이 `[주제] — [구조] (을/를) [동사] [역할]` 패턴
- [ ] **모든 H2 헤더 끝에 내용 특화 이모지 한 개** (H1/H3 이하에는 금지)
- [ ] Why / What / How / Reference 구조 유지
- [ ] Why 가 가정 설정 → 자연스러운 설계 → 문제 발견 → 문제 명명(footnote) → 글의 약속 흐름
- [ ] What 절이 인과 사슬 (앞 절의 미해결 → 뒷 절의 입력)
- [ ] 각 What 절 직후에 인라인 실습 배치
- [ ] How 절이 종합 실습 (개별 실습의 반복이 아닌 조합/역추적)
- [ ] 분할 시 편 간 다리도 정합성 검증

### 톤과 문맥
- [ ] 본문에 비유/과장 표현 없음 (블랙박스, 정체, 실체, 비로소, 드디어)
- [ ] 각 절이 Tension → Resolution 리듬을 따름 (기존 가정 → 뒤집힘 → 실제 구조)
- [ ] 가정 뒤집기가 물 흐르듯 연결됨 (가정 소개 → 왜 합리적인지 → 반례 → 해결)
- [ ] 절 전환이 인과적 ("~만으로는 ~가 해결되지 않는다")
- [ ] 절 순서가 Outside-In (관찰 가능한 현상 → 내부 원인)
- [ ] 코드 블록에 전(맥락)/내(하이라이트)/후(takeaway) 포함
- [ ] 문단당 새 개념 1개 + 즉시 구체화 (정보 밀도 제어)
- [ ] 실습 브릿지에 개연성 ("왜 이 시점에 이걸 해야 하는지")
- [ ] 실습 결과 뒤 개념 회수 (절당 1-2회, 과하지 않게)
- [ ] 본문에 이모지 없음 (H2 헤더 외)
- [ ] 한 문단 한 주제, 문단 내 이어 쓰기, 문단 간 빈 줄 분리

### 보조 요소
- [ ] mermaid 4-5 개 (편당, 전체 구조를 한 눈에 보여줄 때)
- [ ] 중요 개념마다 footnote (편당 최소 20개), 출처 종류 혼합
- [ ] 모든 구조가 마크다운 표 (ASCII 없음)
- [ ] 어려운 용어는 인라인 풀이
- [ ] 파일을 _7단계에서 확정한 경로_ 에 저장
