# /blog-refine -- 기존 기술 블로그 글 개선

대상 파일: **$ARGUMENTS**

`$ARGUMENTS`가 비어 있으면 사용자에게 대상 파일을 묻는다.

---

# 참조 스타일

다음 블로그의 공통 패턴을 기준으로 개선한다.

- **poor.dev/blog** — 짧은 문단, 강한 도입부, 코드 전후 맥락 설명
- **Discord Engineering Blog** — 문제 → 시도 → 결과 내러티브, 데이터 기반 주장
- **Dropbox Tech Blog** — 한 문단 한 아이디어, 명확한 섹션 전환

---

# 워크플로우 (5단계)

## 1단계 — 진단

대상 파일을 읽고 다음을 분석한다.

### 구조 진단

| 항목 | 확인 |
|---|---|
| 도입부가 문제/동기를 명확히 제시하는가 | |
| 각 섹션이 인과적으로 연결되는가 | |
| 코드 블록 전후에 맥락 설명이 있는가 | |
| 결론/요약이 존재하는가 | |
| 헤더 계층이 일관적인가 (h1 → h2 → h3) | |
| Footnote가 충분한가 (최소 5개, 출처 혼합) | |

### 가독성 진단

| 항목 | 확인 |
|---|---|
| 한 문단이 한 아이디어인가 | |
| 문장이 지나치게 길지 않은가 (40자 이상 복문) | |
| 전환문이 자연스러운가 | |
| 독자가 "왜 이걸 읽어야 하는지" 알 수 있는가 | |

### 톤 진단

| 항목 | 확인 |
|---|---|
| 서술체 (다/이다) 일관성 | |
| 비유/과장 표현 | |
| 헷지 표현 (아마, 어쩌면) | |

진단 결과를 **표로 요약**하여 사용자에게 보여준다.

```
## 진단 결과

| 영역 | 점수 | 주요 이슈 |
|---|---|---|
| 구조 | B | 도입부 약함, 결론 없음 |
| 가독성 | C | 문단이 길고 전환 부재 |
| 톤 | A | 일관적 |

### 개선 계획 (N개 항목)
1. ...
2. ...
3. ...

진행할까요?
```

**사용자 승인을 기다린다.**

## 2단계 — 구조 개선

승인을 받으면 구조부터 수정한다.

### 도입부 (첫 h1 또는 첫 단락)

뜬금없이 상황을 던지지 않는다. 논문 서론이나 기술 블로그 intro처럼 문맥을 쌓아 문제를 드러낸다.

다음 흐름을 따른다.
1. **가정 설정**: "~라는 시스템을 설계한다고 가정해보자"
2. **자연스러운 설계**: "이때 다음과 같이 구성할 수 있을 것이다"
3. **문제 발견**: "이 구조에는 큰 문제가 숨어 있다. ~하면 ~가 된다"
4. **문제 명명**: 문제의 공식 명칭과 배경 (footnote로 출처 연결)
5. **글의 약속**: "이 글은 이 문제를 해결하는 N가지 방법을 비교한다"

```
❌ 뜬금없는 시작:
"주문을 DB에 저장하고, 이벤트를 Kafka로 발행하는 코드를 작성한다."

✅ 문맥 있는 시작:
"전자상거래 시스템을 설계한다고 가정해보자.
고객이 주문을 넣으면 주문 서비스는 DB에 주문을 저장하고 결제 서비스에 이벤트를 보내야 한다.
이때 다음과 같이 코드를 작성할 수 있을 것이다. (코드)
이 구조에는 큰 문제가 숨어 있다. ..."
```

### 섹션 간 전환

각 h2 섹션의 끝에 다음 섹션으로의 인과적 전환을 확보한다.

```
❌ "다음으로 X를 본다."
✅ "여기까지 X를 확인했다. 그런데 X만으로는 Y가 해결되지 않는다. Y를 이해하려면 Z가 필요하다."
```

### 결론

글 끝에 핵심 요약이 없으면 추가한다.
- 이 글에서 다룬 것 (2-3문장)
- 독자가 얻어가는 것

## 3단계 — 가독성 개선

### 문단 분리

- 한 문단에 여러 아이디어가 섞여 있으면 분리
- 빈 줄로 단락 구분 (remark-breaks가 줄바꿈 처리)

### 문장 다듬기

- 40자 이상 복문 → 두 문장으로 분리
- 인과 연결 추가 ("~이므로", "~때문이다")
- 목적어 없는 문장 보완

### 코드 블록 맥락

코드 블록 직전에 "무엇을 보여주는 코드인지" 한 줄 추가.
코드 블록 직후에 "핵심 포인트" 한 줄 추가.

## 4단계 — Footnote 추가

원본에 footnote가 없거나 부족하면, WebSearch 등을 활용하여 적합한 참고자료를 찾아 추가한다.

### Footnote 규칙

- 형식: `[^N]: 설명. <https://url>`
- 각 주요 패턴/개념의 첫 등장 시 최소 1개
- 글 전체에 최소 5개, 권장 8~12개
- 출처 종류를 혼합: 공식 문서 / 논문 / 권위 있는 블로그 / 디자인 패턴 사이트

### 배치 기준

- 패턴의 공식 명칭 첫 등장 → 원전 또는 공식 문서
- 핵심 개념 (CAP, ACID 등) → 논문 또는 위키피디아
- 구현체 (Debezium, Dapr 등) → 공식 documentation
- 도입부의 문제 정의 → 해당 문제를 정의한 원전

### 금지

- 일반 상식이나 자체 정의 용어에 footnote 달지 않음
- 접근 불가능한 URL 사용 금지

## 5단계 — 미세 조정

### frontmatter 정리

- description이 비어 있으면 첫 문장 기반으로 작성
- tags가 SSOT(src/domain/tags.ts)에 맞는지 확인

### 불필요한 요소 제거

- 중복 설명
- 의미 없는 전환문 ("자, 그러면...")
- 과도한 강조 (한 단락에 bold/italic 3개 이상)

## 6단계 — 결과 제시

수정된 파일을 저장하고, 변경 요약을 표로 보여준다.

```
## 변경 요약

| 항목 | Before | After |
|---|---|---|
| 도입부 | 바로 설명 진입 | 문제 상황 제시 → 해결 예고 |
| 섹션 전환 | 3곳 전환 부재 | 인과적 전환 추가 |
| 문단 수 | 12 | 18 (분리) |
| 결론 | 없음 | 핵심 3줄 요약 추가 |
| description | 비어있음 | "..." 작성 |
```

# 톤 규칙

`/blog-korean` 의 톤 규칙을 동일하게 따른다.

## 따른다

- `다 / 이다` 체 (서술체)
- 직접적, 서술적 산문
- 한 줄에 한 문장, 문장 사이 빈 줄 한 개

## 금지

- `네요 / 예요 / 죠` 어미
- 독자 직접 호명 (`여러분`, `우리`)
- 비유적/과장 표현 (`마법`, `우아한`, `블랙박스가 아니게 된다`)
- 헷지 표현 (`아마`, `어쩌면`)
- 본문 이모지 (H2 헤더 끝 제외)

---

# 제약

- **원본의 핵심 내용과 주장을 변경하지 않는다** — 구조/흐름/가독성만 개선
- **새로운 기술적 내용을 추가하지 않는다** — 원저자의 지식 범위를 넘지 않음
- **문체를 과도하게 바꾸지 않는다** — 원저자의 개성을 유지하되 가독성만 개선
- 한 번에 전체를 출력하지 않는다 — 진단 → 승인 → 수정
